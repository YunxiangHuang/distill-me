# distill-me — 把专家脑中的隐性知识蒸馏成被代码验证的资产

> 你脑子里那些"只有你知道"的系统机制、设计权衡、踩坑经验，**口述出来默认是不可靠的**——往往不是记性问题，是分层/版本/时间视角差，连你自己都未必察觉偏在哪。
> distill-me 提供一套**经实战校准的蒸馏流程**：把隐性知识变成 ①带代码锚点、已验证的知识库 + ②能改变未来 AI 会话行为的可触发 skill。**这个仓库分享的是流程本身**——fork 它、换上你的领域即可复用。

## 为什么值得做

1. **口述会腐烂，且半对的最危险**。"代码 → 描述"派生出来的断言会随重构漂移；全错的会被识破，**半对的会被直接采信**。蒸馏方向相反——"专家断言 → 代码裁决"，产物是被代码证实的人类判断（含代码里读不出的为什么、事故史、裁决）。
2. **文档不可触发，skill 可以**。一份知识文档躺着不会改变任何 AI 会话的行为；蒸成 skill 后，未来任何会话碰到匹配场景会自动加载。蒸馏的收尾验证（RED→GREEN）就是要证明这个差距：让没有 skill 的 agent 在典型任务上犯错、带 skill 复跑不犯，才算上线。
3. **术语二义性会静默反转业务含义**。"提示用户 refund"——用户是商家还是顾客？答案反了，代码就反了。术语表（ubiquitous language）是蒸馏的强制环节。

## 核心协议（六步）

```
① Scaffold → ② Grill → ③ Verify → ④ Capture → ⑤ Glossary final-align → ⑥ Distill
   搭骨架      逼问专家    代码证实/证伪   只落已验证     术语终审            蒸成 skill
   ├────────── Glossary-first：新名词出现的瞬间过术语表准入，贯穿全程 ──────────┤
   └── Memory loop（有记忆库则必走）：召回@① · 复验@③ · 索引[先删后写]@④ · 对账@⑥ ──┘
```

**三条铁律**（完整版见 [CLAUDE.md](CLAUDE.md)）：

1. **落库前先验证**：专家口述的"机制 / 怎么跑"类断言默认未验证，必须拿真实代码证实 / 证伪；有偏差**回炉再问**，绝不静默取舍（偏差的成因往往本身就是知识）。"为什么 / 业务意图"类专家是权威。
2. **术语先行**：新名词（包括 AI 自创的文件名 / skill 名！）先查重、查组织内既有含义、给定义——之后才允许扩散进任何产物。（实案：一个 AI 自创名撞上组织里既有的同名产品概念，后置发现导致一连串文件 + skill 改名和全仓引用重写；准入时一个问题就能避免。）
3. **知识分层存放**：领域语义（机制 / 不变量 / 术语）→ 蒸馏进 `knowledge/`；架构断言 / 拓扑 / 方向 → 记忆库时点验证条目 + 召回即复验；代码坐标最稳定，作前两层的锚点。

### 1. 建你自己的 distill 仓库

```
your-distill/
├── CLAUDE.md                 # 抄本仓 CLAUDE.md 的"核心 workflow + 三铁律"，换上你的领域
├── knowledge/
│   └── glossary.md           # 从空表开始，第一批词条会在首轮蒸馏中自然产生
├── pending/gaps.md           # 未决项 punch-list（claim · why · what's-needed · owner）
└── .claude/skills/
    └── distill-domain-knowledge/   # 从本仓复制，symlink 到 ~/.claude/skills/
```

### 2. 跑第一个蒸馏会话

对你的 AI agent 说：**"开始蒸馏我自己"**（会触发 `distill-domain-knowledge` skill），然后：

- **选题**：挑一个你最常被问、或新人最常做错的子域。一个子题 ≈ 一个会话。
- **Grill**：AI 会一次一个分支地逼问你，每个问题自带它的推荐答案让你确认 / 纠正——你只需要对错误处开火。说不准的就说"不确定"，会进 gaps 清单而不是被编造。
- **Verify**：你的每条机制断言会被并行 agent 拿真实代码核验。**预期心理建设：会有相当比例被标记 partial**——这不是你记性差，是分层 / 版本 / 时间视角差，被纠正的偏差正是蒸馏的最大价值。
- **回炉**：代码与你说的不一致时，AI 会带着证据回来再问——你的裁决（"已知可接受" / "新发现" / "实际不可达"）也会被记录。

### 3. 验收产物

- `knowledge/<域>/<概念>.md` × N——每条 ✅ 断言带 `路径:行号` 锚点（没有锚点不准标 ✅）；
- `glossary.md` 新词条（含 scope-type 与消歧标记）；
- `pending/gaps.md` punch-list（未决项是活清单，不是沉默的坑）；
- `.claude/skills/<域>/SKILL.md`——**必须过 RED→GREEN**：让一个没有 skill 的 agent 在典型任务上犯错（RED），带 skill 复跑不犯（GREEN），才算上线。

### 4. 日常维护

- 新会话里被召回的旧结论：**时点验证 ≠ 永久真理**，使用前带当前上下文复验锚点；
- gaps 清单的条目在后续蒸馏中顺手收割；
- 修订知识后同步重索引（如果接了记忆库）。

## FAQ

**Q: 和"让 AI 读代码自动生成文档"有什么区别？**
方向相反。自动生成是"代码 → 描述"，会随重构腐烂；蒸馏是"专家断言 → 代码裁决"，产物是**被代码证实的人类判断**——包括代码里读不出来的为什么、事故史和裁决（如"这个缺陷不修，因为……"）。

**Q: 一定要某种特定的代码结构吗？**
不。协议对任何"有专家 + 有代码"的场景成立。结构和纪律是通用的。

**Q: 这套流程自己是怎么校准出来的？**
它是在一个真实生产域上反复打磨出来的（那个域的知识库留在作者的私有仓，不在本仓）。本仓分享的是从那些踩坑里提炼出的**通用部分**——六步协议、三铁律、RED→GREEN 验证法。
